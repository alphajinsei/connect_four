# Connect Four 強化学習プロジェクト

## ユーザーの目標
強化学習（DQN）を勉強中。Connect Four（四目並べ）を題材に、AIが学習して強くなる過程を実感したい。
**最終目標: 初心者の人間より強くなる**

## 学習ロードマップ（現行）

### 現在の方針（2026-03-17 更新）
1. **カリキュラム学習（NoisyRuleBased相手）+ 小スケール中間報酬 + 遅延コールバック方式** ← ゼロから再学習中
   - noise=0.8 → noise=0.5 → noise=0.2 → Phase 4（noise=0.1×50% + スナップショットプール×50%）
   - 昇格条件: vs Noisy 500戦 × 2回連続クリア（まぐれ昇格防止）
   - GameRunner: on_step_endを「相手が1手打った後」に遅延し、next_stateを自分のターンの状態に修正
   - DQN更新式: 通常の `r + γ*maxQ(s',a')`（符号反転なし）
   - 中間報酬は勝敗報酬（±1.0）の1/10以下にスケールダウン

## 設計の重要ポイント

### env/connect4_env.py
- `step()` は相手を呼び出さない（self-playやカリキュラム学習に対応するため）
- `get_state(perspective)` で視点を切り替えられる（DQN訓練・self-playで重要）
- 報酬は常に PLAYER1 視点で返し、`GameRunner` が各エージェント視点に変換する
- 状態は shape `(3, 6, 7)` の float32 配列
  - ch0: 自分のコマ
  - ch1: 相手のコマ
  - ch2: 自分のターンなら 1.0、そうでなければ 0.0

### agents/dqn_agent.py
- 入力: `3 × 6 × 7 = 126`、出力: `7`（列インデックス）
- アーキテクチャ: `Linear(126→256) → ReLU → Linear(256→256) → ReLU → Linear(256→7)`
- NumPyベース（PyTorch不使用）
- ε-greedy + 無効手マスキング、ReplayBuffer、ターゲットネットワーク付き
- `save()`/`load()`: 重みのみ（WebUI・スナップショットプール用、軽量）
- `save_checkpoint()`/`load_checkpoint()`: 重み + ReplayBuffer + ε + total_steps + Adam状態（学習再開用）
  - チェックポイントは `_checkpoint.npz` として別ファイル保存（`np.savez_compressed`で圧縮）
  - `load_checkpoint(path, load_buffer=False)` でバッファのみスキップ可能

### agents/rule_based_agent.py
- 学習なし・手書きルールで動く
- 優先順位: ①自分が即勝てる列 → ②相手が次に勝てる列を防ぐ → ③中央寄りをランダム選択
- ランダムAI相手の勝率: 先手99.5%、後手96%
- 現在のDQNの勝率: 未計測（カリキュラム学習で段階的に目指す）

### agents/noisy_rule_based_agent.py（★新規 2026-03-16）
- `RuleBasedAgent` を継承し、`noise` 確率でランダム手を混入
- `noise=0.0`: 完全ルールベースAI、`noise=1.0`: 完全ランダム
- カリキュラム学習の各フェーズで使用（`train.py` が自動切り替え）

### 現在のハイパーパラメータ（train.py）
| パラメータ | 値 | 備考 |
|---|---|---|
| `lr` | 5e-4 | 1e-3では崩壊が激しかったため下げた |
| `epsilon_end` | 0.10 | 0.05では固定相手への過学習が激しかったため上げた |
| `epsilon_decay` | 0.99990 | ~30000ステップ(≒3750ep)でε=0.10に到達 |
| `warmup_steps` | 2000 | バッファを十分蓄積してから学習開始 |
| `target_update_interval` | 500 | 200では不安定だったため増やした |
| `buffer_capacity` | 50000 | 多様な経験を保持 |
| `batch_size` | 128 | 安定した勾配 |

### GameRunner の遅延コールバック方式（2026-03-17 導入）

**旧方式（廃止）:** 自分が打った直後に `on_step_end(s, a, r, s', done)` を呼んでいた。
- `s'` = 「自分が打った直後の盤面（次は相手のターン）」
- 問題: Q学習の理論的前提が崩れる。`s'` は相手のターンなので、自分がこの状態から行動するわけではない。
  `max Q(s',a')` の予測値は「自分のターンの状態」で学習しているため、「相手のターンの状態」を入力した予測は信頼できない。
- 対症療法として `r - γ*maxQ(s')` と符号反転していたが、本質的には不正確だった。

**現行方式:** `on_step_end` を「相手が1手打った後（再び自分のターン）」に遅延して呼ぶ。
- `s'` = 「相手が打った後、再び自分のターンの盤面」
- 通常のDQN更新式 `Q(s,a) = r + γ * max Q(s',a')` がそのまま使える
- ゲーム終了時は、手を打ったプレイヤーと相手プレイヤーの両方に done=True で確定させる
- 相手側の pending には「中間報酬 + 終局報酬」を合算して返す

### 重要なバグ修正履歴
- **2026-03-16**: `r + γ*maxQ` → `r - γ*maxQ` に修正（当時は next_state が相手ターンだったため）
- **2026-03-17**: GameRunner を遅延コールバック方式に修正し、`r - γ*maxQ` → `r + γ*maxQ` に戻した（next_state が自分のターンになったため、通常の更新式が正しい）

### 中間報酬（シェーピング）設計（2026-03-16 改訂）

**旧設計（廃止）:**
- 2連=±0.05、3連=±0.10のみ
- 問題: 「相手の勝ち手を見逃した」ことへのペナルティがなく、防御を学べなかった

**旧設計（2026-03-16 廃止）:**
| 条件 | 報酬 |
|---|---|
| 自分の3連を作った | +0.30 |
| 自分の2連を作った | +0.10 |
| 相手の3連を防いだ | **+0.50** ← 攻撃より高かった |
| 相手の2連を防いだ | +0.10 |
| 相手の勝ち手を見逃した | -0.80 |

**問題:** Phase 1・2のほぼランダムな相手に対して防御報酬が攻撃報酬より大きいと、「無意味な防御（ランダムな相手は脅威にならない）」が「積極的な攻撃」より得になってしまい、攻撃を学ぶ動機が弱くなる。

**旧設計（2026-03-17 廃止）:**
| 条件 | 報酬 |
|---|---|
| 自分の3連を作った | +0.50 |
| 自分の2連を作った | +0.15 |
| 相手の3連を防いだ | +0.30 |
| 相手の2連を防いだ | +0.10 |
| 相手の勝ち手を見逃した | -0.80 |

**問題:** 勝敗報酬（±1.0）に対して中間報酬が大きすぎた。1ゲーム中に中間報酬が複数回発生するため、累積中間報酬が勝敗報酬を上回り、「勝つ」より「3連を作り続ける」方がQ値的に得になりうる。

**現行設計（2026-03-17 改訂）:**
| 条件 | 報酬 |
|---|---|
| 自分の3連を作った | +0.10 |
| 自分の2連を作った | +0.03 |
| 相手の3連を防いだ | +0.06 |
| 相手の2連を防いだ | +0.02 |
| 相手の勝ち手を見逃した | -0.15 |

**設計思想:**
- **勝敗報酬（±1.0）が圧倒的に大きいことが最重要** — 中間報酬はあくまで「ヒント」
- 攻撃 > 防御の比率は維持（攻撃が直接勝利に繋がるため）
- 盤面全スキャンではなく置いたマス起点のみなので高速

**フェーズ別報酬変更の検討と見送り理由:**
- Phase 1・2（弱い相手）は攻撃重視、Phase 3・4（強い相手）は防御重視、という切り替えも考えたが廃止
- ReplayBufferに前フェーズの経験が残るため、「報酬の意味が途中で変わる」と同じ盤面に対するQ値の正解が変わり学習が混乱する

## 過去の方針と廃止理由

### ステージ4（vs ルールベースAI固定）が失敗した理由（2026-03-16追記）

**固定相手への過学習・崩壊サイクル**

20000epのルールベースAI固定学習でvs RuleBased勝率を測定したところ：
```
ep7500: 33%（ベスト） → ep15000: 0% → ep17000: 19% → 不安定に推移
```
εが0.05に収束した後に「ルールベースAIの癖への過学習」が始まり、その後崩壊するパターンが繰り返された。

**ε=0の実力評価は0%だった**

学習中の評価（ε=0.05混入）では33%に見えたが、ε=0.0（純粋推論）での評価では**0%**だった。
学習中のevalにランダム手（ε）が混じることで「まぐれ勝ち」が数字を水増ししていた。

**ルールベースAIが強すぎた**

ルールベースAI（先手）vs ランダムAI（後手）の勝率は**99%**。
学習初期の弱いDQNには信号が「負け(-1.0)ばかり」になり、「何をすれば良いか」を学べなかった。

**→ 解決策: NoisyRuleBasedAgent（段階的に強くする）+ 攻防セット中間報酬**

### ステージ1・2（ランダムAI相手、カリキュラム学習）が失敗した理由

**根本的な問題: ランダム相手に学習しても意味がない**

ランダムAIはでたらめに動くだけで戦略を持たない。ランダムAIに勝てる戦略は：
- 「適当に並べて4連を狙う」だけで十分
- 相手の3連を防ぐ必要がほぼない（ランダムなので4連まで揃えることが少ない）

結果として、DQNは「攻撃的に並べる」ことは学んでも「相手の攻撃を防ぐ」ことを学ばない。
ルールベースAI（必ず勝ち手を打ち、必ず負け手を防ぐ）相手に試したところ、**勝率0%**だった。

**先手・後手の混在問題**

ステージ1・2では DQN が PLAYER1（先手）とPLAYER2（後手）両方で学習していたが、
Connect Four は**先手必勝ゲーム**のため、先手と後手では最適戦略が全く異なる：
- 先手: 中央を取り積極的に4連を狙う
- 後手: 相手の攻撃を防ぎながら反撃の機会を待つ

混在学習では両戦略が干渉し、どちらも中途半端になっていた可能性がある。

**カリキュラム学習の「過去の自分」問題**

カリキュラムの Phase 2・3 では「過去の自分（DQN）」を相手にしたが、
その過去の自分もランダム相手に学習したものであり、**弱い相手同士で学習した**に過ぎなかった。

### ステージ3（Self-play）が失敗した理由

**破滅的忘却（Catastrophic Forgetting）**

Self-playでDQN同士が対戦すると「対DQN戦略」に特化していき、ランダムAIへの対処能力まで失った：
```
純Self-play: vs Random 77% → 44% に崩壊
```

**混合学習（random-mix）も根本解決にならなかった**

ランダムAIをプールに混ぜて忘却を防ごうとしたが、
そもそも「ランダムAIに勝つ能力」を維持してもゴール（初心者の人間に勝つ）には遠い。
ランダムAIは「目をつぶって打つ相手」であり、実力向上の素材として不適切だった。

## 現行方針の論拠

**「まず攻め方を覚えてから、防ぎ方を覚える」**

人間も将棋・囲碁を覚えるとき、段階的に強くなる。DQNも同様に、いきなり強い相手（ルールベースAI固定）と戦わせても信号が薄すぎて学習できない。

### NoisyRuleBasedAgent（段階的カリキュラム）

`noise`パラメータで相手の強さを連続的に調整できる：
```python
# noise=1.0 → ランダムAIと同じ
# noise=0.0 → 完全なルールベースAI
# noise=0.3 → 70%はルールに従い、30%はランダム
```

カリキュラム設計（昇格条件: vs Noisy 500戦 × 2回連続クリア）:
| フェーズ | 相手 | 目標勝率 |
|---|---|---|
| 1 | NoisyRuleBased(noise=0.8) | 80% |
| 2 | NoisyRuleBased(noise=0.5) | 70% |
| 3 | NoisyRuleBased(noise=0.2) | 50% |
| 4 | NoisyRuleBased(noise=0.1)×50% + スナップショット×50% | 50% |

**昇格条件の設計思想:**
- 旧設計（vs Noisy 200戦 1回クリア）ではまぐれ昇格が発生（ph3を1500epで通過）
- 500戦にすることで振れ幅を±7% → ±4%程度に縮小
- 2回連続クリアを要求することで「一時的な上振れ」での昇格を防ぐ

**Phase 4の設計思想（当初案・実験済み）:**
- 完全ルールベース固定（noise=0.0）では過学習→崩壊サイクルが繰り返された（ステージ4の失敗再現）
- noise=0.1（90%ルールベース、10%ランダム）で微妙なランダム性を加えて癖への過学習を防ぐ
- スナップショット（過去の自分）を50%混ぜることで多様な相手と戦い、特定の癖への過学習を防ぐ
- ベスト更新のたびに新スナップショットをプールに追加（学習と並走してプールが成長する）

**→ Phase 4（プール方式）は失敗した（2026-03-17）**

実験結果:
```
ph4移行直後: vs RuleBased 86〜87%（ph3からの高水準を維持）
ep3000〜: vs RuleBased 18〜58%（急落・不安定）
ep5000〜: vs RuleBased 5〜11%（崩壊）
```

推定原因（2つ）:
1. **プール内の弱い自分が悪影響** — スナップショットにrb8%など弱い重みが混入。弱い相手に勝てる「攻撃的すぎる戦略」が強化され、防御や対強敵戦略が崩れた。rb50%フィルタを導入しても、ph4開始直後にεが不安定なためrb11%の弱い重みがプールに混入してしまった。
2. **過去の自分との対戦自体が問題** — スナップショットはどれもDQNより弱い（過去の自分）。ステージ3のself-play失敗と本質的に同じ構造で、弱い相手との対戦が学習を退化させる。

### 攻防セット中間報酬の狙い

- 攻撃報酬（自分の2連・3連）→ 積極的に4連を狙う動機
- 防御報酬（相手の2連・3連を防いだ）→ 妨害する動機
- 防御失敗ペナルティ（相手の勝ち手を見逃した）→ 防御の失敗に即座に気づかせる

攻撃 > 防御に設定することで「負けないけど勝てない（守備偏重）」を防ぐ。

### 中間報酬はルールベースAIを手書きで教えているのか？

ある程度そうだが、**目的は基礎を身につけること**。基礎を身につけた後に、DQNはルールベースAIを超える戦略（複数の脅威を同時に作るフォークなど）を自然に学べる可能性がある。
